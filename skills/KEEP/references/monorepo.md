# Monorepo support — details and examples

The principle is summarized in `SKILL.md`: one `/knowledge/` at the repo root, with per-package subdirectories *only* under `specs/`. Decisions (ADRs) and ideas stay flat. Runbooks and architecture docs are **specs with a tag**, not separate folders.

This file covers the rationale, the full layout, command behaviors in a monorepo, and the decision tree for cross-package content.

## Why a single zone

Most knowledge in a monorepo is inherently cross-cutting:

- A shared auth library is used by three services. Its spec is one spec, not three.
- The deployment pipeline touches all packages. Its operational behavior is one runbook-tagged spec.
- A decision like *"we use Postgres for everything"* applies wherever data is stored.

Per-package knowledge zones force artificial duplication of all of the above and require the agent to scan multiple zones to answer a single question — exactly what KEEP is trying to avoid. A single root-level zone with package-aware subdirectories under `specs/` keeps the read path simple and lets cross-package relationships emerge through the `related:` field in frontmatter.

## Full layout (v2)

```
/                                          ← repo root
├── knowledge/
│   ├── docs/
│   │   ├── specs/
│   │   │   ├── auth-service/                  ← specs for the auth-service package
│   │   │   │   ├── jwt.md                       (tags: [auth, jwt])
│   │   │   │   ├── jwt-rotation.md              (tags: [auth, runbook])     ← runbook is a TAG
│   │   │   │   └── topology.md                  (tags: [auth, architecture]) ← architecture is a TAG
│   │   │   ├── inference-api/                 ← specs for the inference-api package
│   │   │   │   ├── pipeline.md                  (tags: [inference])
│   │   │   │   └── gpu-oom.md                   (tags: [inference, runbook])
│   │   │   └── shared/                        ← specs spanning multiple packages
│   │   │       ├── api-conventions.md           (tags: [api])
│   │   │       └── overview.md                  (tags: [architecture])       ← cross-cutting topology
│   │   └── decisions/                       ← ADRs always flat, never per-package
│   │       ├── ADR-0001-monorepo-tooling.md
│   │       ├── ADR-0002-auth0.md
│   │       └── ADR-0003-ray-serve.md
│   ├── ideas/                               ← inbox, flat
│   └── INDEX.md                             ← auto-generated by scripts/build_index.py
├── packages/
│   ├── auth-service/
│   ├── inference-api/
│   └── shared/
└── ...
```

Two things to internalize from this layout:

1. **Only `specs/` is package-partitioned.** Decisions and ideas stay flat because their scope is encoded in their content (an ADR's title and body make the affected packages obvious; an idea is half-formed by definition).
2. **`runbook` and `architecture` are reserved frontmatter tags, not directories.** `/keep-ask` filters by tag; the file lives next to the related behavioral spec, which is usually where the reader actually wants it.

## Command behaviors in a monorepo

### `/keep-ask <question>` — the read path

Reads `INDEX.md`, filters by frontmatter `description` / `domain` / `tags`. When the question mentions a package name (e.g. *"how does the inference API handle GPU OOM?"*), the package filter narrows results before tag and domain filters apply. Flat ADRs and `specs/shared/` content surface alongside as cross-cutting context.

A typical answer cites by `id` and groups by scope:

```
The inference-api retries GPU OOM with exponential backoff and reduces batch size
on the second attempt [SPEC-inference-gpu-oom]. This behavior was set when we
adopted Ray Serve [ADR-0003] which doesn't auto-shrink batches at the runtime
level. Cross-cutting: shared retry conventions in [SPEC-shared-api-conventions].
```

If the question doesn't name a package, `/keep-ask` searches across all package subdirectories without grouping.

### `/keep-observe` — diff classification

Infer affected packages from the file paths in the diff:

- `packages/auth-service/src/...` → `auth-service`
- `packages/inference-api/src/...` → `inference-api`
- Both in the same diff → cross-package (and possibly a candidate for `specs/shared/`)

Suggested update paths go under the affected package's subdirectory for behavioral specs and runbook-tagged specs. ADR suggestions always reference the flat `decisions/` directory. Architecture-tagged specs go either under the package (when scope is package-local) or under `specs/shared/` (when scope is cross-cutting).

Example output for a cross-package diff:

```
Detected changes:

Feature (cross-package):
- Updated shared API response envelope — affects auth-service and inference-api

Architecture (cross-cutting):
- The response-flow diagram in specs/shared/overview.md needs the new envelope shape

Suggested knowledge updates:
- knowledge/docs/specs/shared/api-conventions.md   (update body + bump description)
- knowledge/docs/specs/shared/overview.md           (update the response flow section)
```

### `/keep-compile` — applies updates + regenerates INDEX

Writes behavioral specs and runbook-tagged specs under the affected package's subdirectory; writes ADRs flat. When a change spans multiple packages:

- **Prefer `specs/shared/`** for the spec itself. A *"shared API response format"* spec belongs in `specs/shared/`, not duplicated in each package directory.
- **List affected packages in `related:`** using the package's path pattern:
  ```yaml
  related:
    - code:packages/auth-service/**
    - code:packages/inference-api/**
    - test:packages/{auth-service,inference-api}/**/*_test.go matching TestEnvelope*
  ```
- **ADRs that affect multiple packages stay flat.** Their title and content make scope clear; `related:` can list multiple package globs.

After applying updates, `/keep-compile` runs `scripts/build_index.py` from the repo root. The script walks `/knowledge/` and regenerates `INDEX.md` from frontmatter — no per-monorepo logic required.

### `/keep-check-drift` — enforcement on the diff

Uses each file's `related:` patterns to determine if any spec/ADR is affected by the diff. The same matching logic works in monorepo and single-package — package-aware globs in `related:` make the matching scope-correct without any monorepo-specific code path.

### `/keep-govern` — periodic hygiene

Runs across the whole zone. Output groups results by package subdirectory under `specs/`, and lists flat artifacts (ADRs, ideas) separately:

```
Stale knowledge detected:

specs/auth-service/:
- knowledge/docs/specs/auth-service/api-keys.md  (last touched 11 months ago; related code changed 4 times)

specs/inference-api/:
- (none)

decisions/ (flat):
- knowledge/docs/decisions/ADR-0003-ray-serve.md  (related code churned this quarter — re-read recommended)

ideas/:
- knowledge/docs/../ideas/2025-11-04-distributed-tracing.md  (in draft for 192 days — promote or deprecate)
```

## Decision tree — where does this content go?

| Content describes... | File location | Tags on the file |
|---|---|---|
| A single package's behavior | `specs/<package>/<feature>.md` | `[<domain>]` |
| Behavior shared by multiple packages | `specs/shared/<topic>.md` | `[<domain>]` |
| Failure modes / on-call procedure for one package | `specs/<package>/<failure-mode>.md` | `[<domain>, runbook]` |
| Failure modes that cross packages (e.g. shared DB exhaustion) | `specs/shared/<failure-mode>.md` | `[<domain>, runbook]` |
| Internal topology of one package | `specs/<package>/topology.md` | `[<domain>, architecture]` |
| Cross-package topology (how packages relate) | `specs/shared/overview.md` | `[architecture]` |
| A decision (any scope) | `decisions/ADR-NNNN-<slug>.md` | — (ADRs use `type: adr`) |
| A half-formed proposal | `ideas/<YYYY-MM-DD>-<slug>.md` | `[<domain>]` |

When the right location is genuinely unclear, `/keep-compile` asks rather than guessing:

> *"This spec describes how the API responds to malformed JSON, which applies to both auth-service and inference-api. Put it in `specs/shared/` (treat as cross-cutting) or in `specs/auth-service/` since that's where the change originated?"*

## `INDEX.md` in a monorepo

`INDEX.md` is **auto-generated by `scripts/build_index.py`** — never hand-authored. The script reads frontmatter from every file in `/knowledge/` and produces a table-based index. In a monorepo, the script automatically groups behavioral specs by their `specs/<package>/` subdirectory; ADRs, ideas, and `specs/shared/` get their own sections.

The generated structure (example):

```md
# /knowledge INDEX  (auto-generated, do not edit)

Generated: 2026-05-15T08:00:00Z by scripts/build_index.py

## Packages (specs/)
- auth-service — 4 specs (2 behavioral, 1 runbook, 1 architecture)
- inference-api — 3 specs (2 behavioral, 1 runbook)
- shared — 2 specs (1 behavioral, 1 architecture)

## All specs
| id | title | package | domain | tags | status |
|----|-------|---------|--------|------|--------|
| SPEC-auth-jwt | JWT validation | auth-service | auth | auth, jwt | accepted |
| SPEC-auth-jwt-rotation | JWT secret rotation | auth-service | auth | auth, runbook | accepted |
| SPEC-inference-pipeline | Inference pipeline | inference-api | inference | inference | accepted |
| ... | | | | | |

## Decisions (flat)
| id | title | supersedes | status |
|----|-------|------------|--------|
| ADR-0001 | Postgres over Mongo | — | accepted |
| ADR-0014 | Ray Serve for inference | ADR-0007 | accepted |
| ADR-0007 | KServe for inference | — | superseded |

## Ideas (flat)
| id | title | created | status |
|----|-------|---------|--------|
| IDEA-2026-05-14-grace-window | JWT grace window during rotation | 2026-05-14 | draft |
```

The headings, the package grouping, the supersession column, the status filter — all derived from frontmatter. If `INDEX.md` ever drifts from this shape, run `scripts/build_index.py` again; if you find yourself wanting to add a section, change the script and regenerate, not the file.

## Single-package fallback

In a non-monorepo, the structure collapses naturally:

- `specs/<package>/` becomes `specs/<area>/` — same convention, "area" instead of "package" (e.g. `specs/auth/`, `specs/payments/`).
- `specs/shared/` becomes unnecessary; cross-area concerns live wherever the dominant concern lives, with `related:` linking the rest.
- The `## Packages` block in `INDEX.md` collapses to a `## Domains` block; the script handles this automatically.

All command behaviors apply unchanged. The monorepo conventions add complexity only where complexity exists in the repo.
